# Consta for Community (SLA)

Последнее обновление: 09.06.2021 г.

Настоящее соглашение определяет политику взаимоотношений при использовании дизайн-системы Consta.

## 1. Лицензия

Дизайн-система Consta и все ее зависимые компоненты распространяются под открытой лицензией [MIT](https://opensource.org/licenses/MIT), а текст лицензионного соглашения дублируется непосредственно в самих репозиториях.

## 2. Стратегия версионирования

### 2.1 Схема версионирования

Во всех зависимых компонентах экосистемы Consta применяется стратегия [семантического версионирования](https://semver.org/lang/ru/).

Использование дополнительных постфиксов для предварительных релизов и сборок с мета-обозначениями допускается, но сохранение в них обратной совместимости не гарантируется, поскольку эти версии не являются конечными.

### 2.2 Жизненный цикл версий

Основной фокус делается на поддержку и стабилизацию крайней major-версии, а пересмотр предложений, которые потенциально нарушают обратную совместимость, допускается не чаще чем раз в полгода для отдельно взятого компонента. Отчетный период начинается с даты последнего релиза.

### 2.3 Механизм оповещений

2.3.1 Для оперативного информирования о новых версиях ведется [канал в Telegram](https://t.me/consta_ui_releases).

2.3.2 Для получения сводной информации обо всех изменениях в программном коде рекомендуется использовать механизм [наблюдения](https://github.com/watching) за проектом.

2.3.3 В случаях, когда сохранение обратной совместимости невозможно в силу наличия потенциальных уязвимостей или серьезных проблем с производительностью применяется механизм [deprecated](https://docs.npmjs.com/cli/v7/commands/npm-deprecate).

### 2.4 История версий

Полная история изменений по всем версиям продуктов доступна на [страницах](https://github.com/consta-design-system/uikit/releases) с релизами.

## 4. Системные ограничения

### 4.1 Браузеры

Ориентируйтесь на то, что все основные компоненты дизайн-системы протестированы и корректно отображаются в браузерах на движке Chromium.

### 4.2 Разрешения экранов

Разрешение экранов, которые смело можно учитывать при проектировании интерфейсов для полноразмерных устройств: 1366×768, 1280×720, 1440×900, 1920×1080.

## 5. Типы тикетов и порядок их заведения

Дефекты, запросы на новую функциональность или вопросы оформляются в виде [тикета на Github](https://github.com/consta-design-system/uikit/issues) или письмом на [design@gazprom-neft.ru](design@gazprom-neft.ru).

Возможные типы тикетов:

| Тип тикета  | Расшифровка                                 |
| ----------- | ------------------------------------------- |
| bug         | Дефект                                      |
| design      | Задача на дизайн                            |
| docs        | Доработка или добавление документации       |
| feature     | Новый функционал                            |
| improvement | Улучшение                                   |
| question    | Вопрос                                      |
| research    | Исследование или анализ                     |
| test        | Задачи на тестирование или покрытие тестами |
| charts      | Задача относится к библиотеке графиков      |
| widgets     | Задача относится к библиотеке виджетов      |
| duplicate   | Дублирующаяся задача                        |

[Правила оформления](https://consta-uikit.vercel.app/?path=/docs/common-develop-issues--page) тикетов:

- При создании тикетов с типом bug необходимо использовать шаблон оформления баг-репорта (Bug report).
- При создании тикетов с типами feature или improvement необходимо использовать шаблон запроса фичи или улучшения (Feature request).

## 6. Время реагирования

В течение трех рабочих дней с момента появления нового тикета применяется следующий алгоритм:

- Если тикет понятен, и вопросов нет, ему присваивается тип и приоритет.
- Если для правильной категоризации тикета требуются уточнения (обсуждение происходит в комментариях), то тип и приоритет присваиваются после получения ответов.

### 6.1 Как определяется приоритет

На приоритет тикета влияют следующие факторы:

- Потребность в проектах ПАО «Газпром нефть».
- Экспертное мнение команды Consta.
- Популярность компонента в других системах.
- Высокая активность пользователей.
- Подробное описание с понятными формулировками проблем или предложений.

- Если за реализацию берется внешняя команда, то сроки реализации могут отличаться.
- Если реализация задачи требует предварительного исследования или дизайна, то создается отдельная задача с меткой research или design.

## 8. Актуализация бэклога

Актуализация бэклога и пересмотр приоритетов всех тикетов происходит один раз в квартал. По итогам пересмотра приоритет тикета может быть как поднят, так и понижен.

### 8.1 Причины закрытия тикетов

В рамках актуализации бэклога тикет может быть закрыт по следующим причинам:

- Дубликат.
- Задача уже реализована.
- Входит в другой тикет.
- Задача потеряла актуальность.
- Не получены ответы на уточняющие вопросы или неполное описание.
- Обоснованный отказ команды Consta.

Задачи имеют право закрывать пользователи с правами triagle и выше.

## 9. Запросы на принятие изменений (PR)

Изменения и доработки, предлагаемые контрибьютерами, [оформляются](https://consta-uikit.vercel.app/?path=/docs/common-develop-contributors--page) в виде [запросов на принятие изменений](https://github.com/consta-design-system/uikit/pulls).

При появлении нового PR команда Consta в течение двух рабочих дней проводит [ревью](https://consta-uikit.vercel.app/?path=/docs/common-develop-review--page) и, при отсутствии критичных замечаний, принимает PR.

К мержу допускаются PR, получившие не менее двух аппрувов. Изменения включаются в следующий релиз. Релизы выпускаются еженедельно по средам.
