# Дизайн-система Consta

Дизайн-система «Газпром нефти». Распространяется бесплатно по лицензии MIT, подходит для создания цифровых продуктов: веб-сервисов и не только.

## Куда смотреть

[Figma](https://www.figma.com/@consta)

Здесь — песочница и документация.

Справочная информация и контакты — на [сайте дизайн-системы](http://consta.gazprom-neft.ru)

## Общие принципы

Дизайн-система предоставляет надёжный, консистентный и прозрачный набор правил, ограничений и инструментов для создания цифровых проектов и продуктов. Облегчает совместную работу разным командам.

**Дизайнеры** смогут быстрее создавать прототипы цифровых продуктов или промо-материалов из готовых компонентов.

**Аналитики** смогут быстрее набрасывать ваерфреймы и лучше анализировать готовый интерфейс.

**Фронтенд разработчики** смогут переиспользовать и дополнять базовые компоненты, не тратя время на написание собственной реализации, и лучше сфокусироваться на уникальных блоках.

**Менеджеры** заранее получат представление о внешнем виде продуктов и ограничениях, смогут тестировать продуктовые гипотезы.

**Тестировщики** получат эталонные компоненты, которые уже протестированы.

**Новые сотрудники** быстрее погружаются в контекст работы.

## Основа дизайн-системы

### БЭМ

Один из ключевых принципов при разработке дизайн-системы — БЭМ-методология. Подробнее про неё можно почитать на [официальном сайте БЭМ](https://ru.bem.info/).

### Whitepaper

В основе дизайн-системы — библиотека [Whitepaper](https://whitepaper.tools). Одна из основных идей Whitepaper — в дроблении интерфейса на 4 визуальных слоя. Такое деление позволяет гибко манипулировать небольшими простыми сущностями. Чтобы сделать что-то сложное и большое, можно просто эти сущности комбинировать. Так мы получаем неограниченное количество комбинаций, которые легко настраиваются независимо друг от друга.

1. **Тема**
   Это первый и самый главный слой интерфейса, в котором описаны правила использования цветов, отступов, зафиксированы семейства шрифтов и их размеры. [Подробнее о темах](?path=/docs/components-theme--playground)
2. **Каркас**
   Описывает верхнеуровневую обвязку страницы и внутренние сетки. В дизайн-системе Consta нет своих уникальных решений для слоя Каркас, используются стандартные решения из Whitepaper. Их можно дополнить, если это нужно для проекта.
3. **Паттерн**
   Блок, описывающий контентную структуру со всеми возможными модификациями. Новые паттерны не создаются, а используются те, что предлагает Whitepaper: карточка, список, таблица, иконка с текстом, форма и другие.
4. **Контент**
   Последний слой описывает конечный контент интерфейса: текст, контролы, графика, иконки и прочее. Это слой больше всего представлен в нашей дизайн-системе.

## Технологии

Сейчас библиотека компонентов дизайн-системы существует только в React-реализации.

- React
- Typescript
- PostCSS ([postcss-nested](https://github.com/postcss/postcss-nested), [postcss-color-function](https://github.com/postcss/postcss-color-function), [postcss-simple-vars](https://github.com/postcss/postcss-simple-vars) для media-query)
- whitepaper.tools

Библиотека рассчитана на браузеры семейства Chromium. Если нужен fallback для IE, есть два решения:

1. Форк от библиотеки с фоллбэками.
2. Собственная реализация по методологии дизайн-системы.

## Особенности

- Определение темы происходит на верхнем уровне DOM-дерева и дотягивается до всех дочерних узлов (iframe не подходит).
- Переопределение темы может происходить на любом узле DOM-дерева.
- Абсолютно все блоки на странице должны использовать переменные из темы (custom-properties). Минимально: все значения цветов, отступов (внутренних и внешних) и размеров типографики должны браться из темы.
- Использование переменных по их смыслу. Например: переменная `--color-typo-link` не должна использоваться для фона блока, а переменная `--color-bg-warning` должна использоваться только для отображения статуса, а не просто раскрашивания фона в рыжий.
